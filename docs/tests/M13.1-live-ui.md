## ✅ Happy Path

| # | Step | Expected |
|---|------|----------|
| H1 | `curl -fsS http://api.agentmesh.localhost/actuator/health` | `200 {"status":"UP"}` |
| H2 | `wscat -c ws://api.agentmesh.localhost/ws` | First frame: `{"type":"connection.established",...}` within **500 ms** |
| H3 | Send `{"type":"ping"}` | Receive `{"type":"pong","timestamp":"..."}` within 100 ms |
| H4 | `POST /api/workflows/start` body `{"projectName":"…","srs":"…","tenantId":"default"}` | HTTP 200, response carries `id` (= workflowId) + `currentPhase=PLANNING` |
| H5 | UI dashboard auto-subscribes (`subscribe.workflow`) | Within **500 ms** of every backend phase change, a `workflow.update` frame arrives at the browser |
| H6 | Phase progression PLANNING → ARCHITECTURE → CODE_GENERATION → TESTING → REVIEW → DEPLOYMENT | **5 paired** `agent.status` frames per agent (`RUNNING` then `IDLE`) |
| H7 | Each agent posts to Blackboard | `blackboard.update` frames flow with `entryId`, `type`, `agentId`; UI timeline grows append-only |
| H8 | Workflow completes | Final `workflow.update` with `status=COMPLETED`, `progress=100`, `phase=DEPLOYMENT` |
| H9 | UI ReactFlow nodes | Color transitions match phase status; latency from backend `lastUpdatedAt` to DOM paint < **500 ms** (Protocol §4 acceptance) |

**Pass criteria:** all 9 steps green, no dropped frames, `connection.established → COMPLETED` round trip reproducible 5/5.

## ⚠️ Edge Cases

| # | Step | Expected |
|---|------|----------|
| E1 | Open 10 concurrent browser tabs subscribing to the same workflowId | All 10 receive every `workflow.update` (broadcast fan-out via `workflowSubscriptions`) |
| E2 | Subscribe to a workflowId that does not exist | No frames; UI shows "waiting…" but never crashes; no server stack trace |
| E3 | Send malformed JSON `not-json` | Server logs error at `WARN`, connection stays open (validated by sending `{"type":"ping"}` next and receiving `pong`) |
| E4 | Kill `agentmesh-api` mid-workflow, bring it back within 30 s | `live-stream.ts` reconnects with exponential backoff (1s → 2s → 4s … capped 15s) and **re-issues `subscribe.workflow`** for all known IDs |
| E5 | MAST violation fires (`POST /api/mast/violations`) | UI receives a `mast.violation` frame and renders a toast; previously cached violations list grows by exactly 1 |
| E6 | Blackboard burst (100 entries in <1 s) | UI keeps newest 100 (`slice(0, 100)`), no UI freeze, frame rate ≥ 30 fps |
| E7 | Subscribe then unsubscribe rapidly 50× | No memory leak: `workflowSubscriptions` size returns to 0 (verify via `/actuator/metrics` or thread dump) |

## ❌ Failure Modes

| # | Step | Expected |
|---|------|----------|
| F1 | Stop `agentmesh-api` entirely | UI badge flips to "disconnected" within 2 s; reconnect attempts visible in console; no uncaught exceptions |
| F2 | Block port 80 on Traefik | UI surfaces connection error; backend logs no leaks |
| F3 | Workflow execution throws inside an agent | Backend marks workflow `FAILED`; final frame is `workflow.update` with `status=FAILED, message="Workflow failed: ..."` |
| F4 | Cancel running workflow (`POST /api/workflows/{id}/cancel`) | Next phase transition halts; UI stops receiving new `agent.status` running frames; `workflow.update` shows `status=CANCELLED` |
| F5 | Database disconnect during `updateWorkflowPhase` | Spring transaction rollback; `webSocketHandler.broadcast*` is **NOT** called (broadcast is post-commit via the transactional method body) — verify by tailing `agentmesh.log` and checking the UI does not see ghost transitions |
